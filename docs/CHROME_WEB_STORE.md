# Publishing Peekly to the Chrome Web Store

A complete walkthrough from "I have a working dev build" to "users are auto-updating from the store". Pair this with [RELEASING.md](../RELEASING.md), which covers the GitHub Releases side.

## One-time setup

### 1. Pay the developer fee

Open the [Chrome Web Store Developer Dashboard](https://chrome.google.com/webstore/devconsole) and sign in with the Google account you want as the publisher.

You'll be prompted to pay **$5 USD**, lifetime. This is Google's anti-spam measure. After payment, the dashboard becomes available.

### 2. Decide on the publisher name

In the dashboard, go to **Account → Publisher info**. Choose how the publisher will appear on the public listing:

- A personal name (e.g. `Romario Sobreira`)
- A group / organization (requires a Google Workspace group)

This shows up under the extension name in search results, so pick something the audience will trust.

## Asset preparation

The store listing has stricter requirements than the manifest. Prepare these before you start the form:

### Icons (already in repo)

| Size | Source | Path |
|---|---|---|
| 128×128 PNG | `public/icons/128.png` | Generated by `bun run gen:icons` |

### Screenshots (required, minimum 1, max 5)

- **Format**: PNG or JPEG
- **Size**: 1280×800 or 640×400 (1280×800 strongly preferred for retina displays)
- **Subject**: actual UI of the extension in use, on a real page

Suggested set for Peekly:

1. **Hero**: a React app (e.g. localhost dev) with the indigo highlight + label `<HeaderSection · <span>>` clearly visible
2. **Big panel open**: shows source card, props, navigation chips, owner chain
3. **Tooltip with all four tabs visible** (cycle through and pick the most informative)
4. **Class breakdown / computed styles** sections expanded
5. **Settings popup** showing the three toggles

### Promotional tile (optional, recommended)

- **Small promo tile**: 440×280 PNG/JPEG. Used in the store search and the "More extensions" carousel.
- **Marquee**: 1400×560 PNG/JPEG. Only used if Google selects you for "Featured" status. Optional.

The tile should be visual (a screenshot crop with the logo), not text-heavy.

### Logo for the listing

Many publishers reuse their 128×128 manifest icon — that works. If you want a higher-resolution variant for the marquee, render `public/icon.svg` at 1024×1024 with the same `bun run gen:icons` pattern.

## Permission justifications

When you submit, the form asks for a justification for each "sensitive" permission. Reviewers actually read these.

### Single purpose statement

```
Inspect React components and DOM elements on web pages from a floating overlay, with one-click jump to the source file in the user's code editor.
```

### `host_permissions: <all_urls>`

```
Required to inject the developer inspector on any site the user chooses to use it on. The picker is OFF by default on every site except localhost / 127.0.0.1 / *.localhost. Users explicitly enable it via the popup toggle for any other site. The extension reads page state (DOM, React fiber tree, CSS) but never transmits it anywhere — there are no network requests of any kind. See PRIVACY.md and the source under src/.
```

### `storage`

```
Stores three user preferences in chrome.storage.local: a global "active" toggle, an "auto-on localhost" toggle, and the user's chosen editor for the "Open in editor" button. No personal data, no telemetry, no remote identifiers.
```

### `activeTab`

```
Activates the inspector on the user's currently focused tab when triggered from the extension popup. Required so the popup toggle can take effect immediately on the page the user is looking at.
```

### Remote code use

```
None. The extension does not load, evaluate, or execute any remote code. All scripts are bundled at build time. The only outbound action is opening a tab with a `vscode://`, `cursor://`, `webstorm://`, `idea://`, `pycharm://`, or `subl://` URL when the user clicks "Open in editor" — these are OS protocol handlers, not network requests.
```

## First submission

1. **Build the package locally** (or download from a GitHub Release):

   ```bash
   bun install
   bun run gen:icons
   bun run zip
   # → ./peekly.zip
   ```

2. **Open the dashboard** → **+ New Item** → upload `peekly.zip`.
3. After upload, Chrome assigns the extension a permanent **Item ID** (32-character lowercase string). **Save this** — you'll need it for CI publishing later, and it's how users will find your extension via direct URL: `https://chromewebstore.google.com/detail/<id>`.
4. Fill out the listing using the drafts above.
5. Upload screenshots and the optional promo tile.
6. Hit **Submit for review**.

### Review timeline

- **First submission**: typically 1–3 business days. Sometimes 1–2 weeks if you have many sensitive permissions or your description is vague. `<all_urls>` will get scrutinized — the justification text above is calibrated for that.
- **Updates**: usually under 24 hours, often within a few hours.

### Common rejection reasons

| Reason | What it means | Fix |
|---|---|---|
| **"Single purpose violation"** | Reviewers think your extension does multiple unrelated things | Sharpen the description and the single-purpose statement to one sentence |
| **"Excessive permissions"** | A permission isn't justified | Either remove the permission or strengthen the justification |
| **"Missing privacy policy"** | URL not reachable | Make sure PRIVACY.md is in the default branch and the URL resolves |
| **"Description doesn't match functionality"** | Mismatch between listing and actual behavior | Test that every feature mentioned in the description works in a fresh install |
| **"Code obfuscated"** | Reviewers can't read the bundled code | Don't ship minified-only builds without sourcemaps or the original on GitHub. Peekly's code is open source — link to the repo in the description |

## Updates after first publish

The release pipeline already handles this end-to-end. See [RELEASING.md](../RELEASING.md) for the local script.

To enable **automatic publish on tag push**:

1. Get OAuth credentials per the [chrome-webstore-upload-keys guide](https://github.com/fregante/chrome-webstore-upload-keys). You need:
   - `client_id`
   - `client_secret`
   - `refresh_token`
2. Add four GitHub repository secrets (`Settings → Secrets and variables → Actions → New repository secret`):
   - `CWS_EXTENSION_ID` (from step 3 of "First submission")
   - `CWS_CLIENT_ID`
   - `CWS_CLIENT_SECRET`
   - `CWS_REFRESH_TOKEN`
3. The `Publish to Chrome Web Store` step in `.github/workflows/release.yml` will activate automatically on the next tag push (it's gated on `CWS_EXTENSION_ID` being set).

From then on, `bun run release:patch` (or minor / major) followed by the two `git push` commands publishes simultaneously to GitHub Releases **and** the Chrome Web Store.

The CWS upload publishes the new version directly. The store re-reviews each update; the extension goes "Pending review" briefly and rolls out when approved.

## Yanking a published version

You cannot delete a specific version from the Chrome Web Store after it ships to users. You can:

- **Unpublish the entire extension** (dashboard → Item → Unpublish). Existing installs continue to work but no new installs and no auto-updates.
- **Push a hotfix version** that supersedes the bad one (the recommended path).
- For a critical security issue: contact Google's [Web Store team](https://support.google.com/chrome_webstore/contact/developer_violations) directly. They can fast-track removal in extreme cases.

## Going from `<all_urls>` to fewer permissions (future)

Reviewers (and users) prefer narrowly-scoped extensions. A future version of Peekly could:

- Drop `<all_urls>` from `host_permissions` and rely on `activeTab` + `scripting.executeScript` invoked from the popup. This means the picker only attaches when the user explicitly clicks the icon, but the install prompt is much less scary.
- Trade-off: loses the auto-on-localhost behavior unless we add a setting that asks for the broader permission only when toggled on.

This is on the roadmap. For v0.x we accept the broader scope and document it.
